# 测试用例文档(test-cases)

> 描述**测试意图**(给人读,也给 agent 读);agent 据此生成 `auto-tests/`(如引入)可执行脚本,对部署环境黑盒实测。

## 两个维度

| 维度 | 路径 | 放什么 | 生成到 |
|------|------|--------|--------|
| **前端流程 e2e** | `web/<流程>.md` | 用户在 web 上走的完整流程(主题切换…),用户视角 | e2e(Playwright) |
| **后端服务契约** | `<服务>/<feature>.md` | 单服务的 HTTP/WS 契约、集成,后端视角 | api / 集成 |

> **为什么分两维**:线上验收主要走前端;一个前端流程常跨多个后端服务。前端流程放 `web/`,后端契约放 `<服务>/`(kebab:`api`、`ws`,与 `services/cmd/<服务>` 对齐)。

## 🔴 跨端规则
后端服务改动若**影响线上用户可感知流程**,必须**连带**更新 `web/` 下对应流程的 e2e 用例,不能只更后端契约。

## TC 格式
每条 TC:前置条件 / 操作步骤 / 预期结果 / 类型(`e2e|integration|api`)/ 优先级(`P0|P1|P2`)。

## 现有骨架
- `web/theme-toggle.md`:前端主题切换流程 e2e 示例。
- `api/`、`ws/`:后端契约骨架(按需新建 `<feature>.md`)。
