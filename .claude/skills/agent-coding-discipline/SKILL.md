# Skill: Agent Coding Discipline(写码 agent 的行为纪律)

> **前提认知**:语言模型在写代码时**会犯可预测、重复性的错**——它快速产出看起来合理的代码,慢于注意到"合理"不等于"正确"。**可靠性来自下面的过程纪律,而不是寄希望于模型自己做对。**
>
> 改编自 Andrej Karpathy《CLAUDE.md: Field Notes on Getting a Language Model to Write Code You Will Not Rewrite》。规则随模型迭代而修订。

---

## 1. 读再写(Read before you write)

- 改一个文件前,**先读**——读,不要扫。
- **复制代码库里已有的模式**;看 imports 弄清这个项目实际依赖什么(别在全用 `fetch` 的地方塞 `axios`)。
- 找不到可以照搬的模式时,**问,不要猜**。

## 2. 想再做(Think before you code)

- 动键盘前先想清楚要做什么。**明示假设**——"加鉴权"有五种意思,说出你选的那一种;**点出 tradeoff**。
- 真的搞不清就停下来问,而不是用看起来合理的代码填空。**那种代码恰恰能通过随手 review、却在关键时刻爆掉。**

## 3. 简单性(Simplicity)

- 写**眼前问题**所需的最少代码,而不是"未来所有变种都能搞"的最少代码。
- 抵制过早抽象、跳过不可能发生的错误处理、把值硬编码到真有理由配置化之前。
- **判定标准**:如果某处抽象的唯一理由是"以防我们以后需要",你已经过度构建了。

## 4. 外科手术式改动(Surgical changes)

- diff 应该和任务一样小。
- **不要碰你没被要求碰的地方**,匹配现有风格,**不要 reformat**——一次 formatter 跑会把真正重要的三行埋进三百行无关变更里。
- **判定标准**:每一行改动你都能用任务来解释。如果某行的存在理由是"反正我都进来了顺手...",**revert 掉**。

## 5. 验证(Verification)

- "能跑的代码"与"你以为能跑的代码"之间的鸿沟,是**测试**。
- 修 bug 时,**先写失败的测试,看它失败,再修**——这是证明你修了**根因**而不是**症状**的唯一方法。
- 测**真正会坏的行为**,而不是"构造器是否设置了某个字段"。
- 如果某段代码**难以测试**,那是关于**设计**的信息,**不是跳过测试的许可**。

## 6. 目标驱动(Goal-driven execution)

- 每个任务**写代码前**都要有**成功标准**。
  - "加校验"变成"给定缺失或格式错误的 email 时,返回 400 + 清晰错误,两种情况都有测试"。
- 多步任务**先说计划**,让人在你花一小时建错方向之前能拦住。

## 7. 调试(Debugging)

- 出问题时,**调查,不要猜**。
- **完整读**报错和 stack trace,**复现** bug 之后再改任何代码,**一次只改一处**。
- 不要用 null check 盖住一个意外的 null;搞清楚**为什么**它是 null,否则 bug 只是搬到了一个更安静的地方。

## 8. 依赖(Dependencies)

- 每个依赖都是**你不掌控的永久代码**。
- 加一个前,先问项目或标准库是否已经能做(`crypto.randomUUID()` 优先于 `uuid` 包)。
- 真要加时,**说明为什么**,让选择可见,而不是悄悄塞进 manifest。

## 9. 沟通(Communication)

- 说**你做了什么、为什么**,而不是甩一堆代码。
- 即使你完全按要求做了,**也要主动指出顾虑**。
- **精确表达不确定**:"我不确定这个库支持流式" 告诉人去验证什么;"我觉得应该能行" 不告诉人任何东西。

## 10. 持续推进(Continuous progress)

- 小步、测试先行、每步可验证——但**"可验证"不等于"每步都停下来等人"**。没遇到**真正需要人工决断**的问题之前,**不中断、一直往下推**。
- 只在这些点停:需求/方案有歧义、要在互斥选项间抉择;或触及不可逆/外发/安全边界(删数据、发外部、动 prod)等只有人能拍板的事。
- **并行时:只有真正卡到人工决断的那个子 session 停下来等人;其余不需要中断的子 session 照常继续开发**——别因一个 sub-agent 卡住就把整批停掉。
- 反面(= 空停,禁止):做完一小步就问"要不要我继续?";把能自己定的小选择(命名 / 默认值 / 等价实现)抛给人。凡是能从需求、代码、合理默认推出的,直接做下去。

## 4 个常见失败模式(发现自己掉进去,**停下来,不要硬上**)

| 名字 | 症状 | 对照规则 |
|---|---|---|
| **Kitchen Sink** | "顺便"重构半个代码库 | → 规则 4 |
| **Wrong Abstraction** | 还没复制粘贴过两次就开始抽象 | → 规则 3 |
| **Optimistic Path** | 只处理 happy path,无视 500 | → 规则 5 |
| **Runaway Refactor** | 一个 fix 级联到多个文件 | → 规则 4 |

## Pre-submit 自检清单(开 PR 前对自己问一遍)

- [ ] 我**读了**我在改的文件,并复用了现有模式吗?(规则 1)
- [ ] 我**先说了假设和计划**而不是直接动手吗?(规则 2、6)
- [ ] 这是**解决问题的最小改动**吗?有没有过度抽象?(规则 3)
- [ ] 每行 diff 都能被任务解释吗?有没有顺手编辑或 reformat?(规则 4)
- [ ] 改动**被测试覆盖**了吗?bug 修复有没有**先写失败的测试**?(规则 5)
- [ ] 每个新增依赖都**有理由**吗?标准库能不能做?(规则 8)
- [ ] 我**说明了做了什么、为什么、有什么顾虑**了吗?(规则 9)
- [ ] 我有没有掉进上面任一个失败模式?(4 个 mode)

> 这份清单不是仪式——任意一条答不出"yes",**停下来补**,再开 PR。
> 它会作为 `.github/pull_request_template.md` 自动出现在每个 PR 描述里。

---

## 给评审者(verifier-*)的指引

verifier-quality 评审时,**显式扫这 4 个失败模式**,见到就 BLOCK:

- Kitchen Sink → 看 diff 文件数 vs 任务范围,扩散了就 BLOCK
- Wrong Abstraction → 看是否新增了"只被调用一次"的抽象
- Optimistic Path → 看错误处理覆盖度
- Runaway Refactor → 看是否一个 bug fix 改动 > N 个文件,且改动间无直接因果

## 反模式(BLOCK 自己)

- ❌ 把 10 条规则当作"建议",真写起来跳过——规则就是给"赶时间想跳过"那一刻准备的
- ❌ 让评审 agent / 同事替你过 checklist——那是放弃**写码者第一责任**
- ❌ 因为 deadline 紧把 fail-first 测试跳过——bug 永远会回归,你只是把它推到下次冲刺
