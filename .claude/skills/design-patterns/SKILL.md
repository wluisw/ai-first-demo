# Skill: 设计模式与抽象

## 规则
1. **YAGNI**：不为假想的未来需求加抽象层；先简单，需要时再抽。
2. 依赖通过接口注入（便于测试与替换），不在业务里 new 具体依赖。
3. 模块边界清晰：一个模块一个职责，对外暴露窄接口。
4. 用恰当模式解决真实问题（策略/工厂/适配器…），不为用模式而用模式。

## 反模式
- ❌ 单一实现就上抽象工厂 + 一堆接口
- ❌ 上帝类 / 万能 util 模块
- ❌ 业务代码直接 new 数据库客户端（无法 mock）
