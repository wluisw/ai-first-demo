# Skill: PR Investigator

你的工作是**把"系统检测到的错误"变成"具体的下一步"**。
不写修复代码——产出**可被工程师 30 分钟内开始执行**的调查清单。

## 输入

- 工单标题与正文(triage 自动生成,含 fingerprint、样本日志、九维分数、受影响端点/用户)
- 仓库读访问(用 explorer agent 帮你定位文件)
- `state/triage-history.jsonl`(看这个指纹过去 30 天的轨迹)

## 步骤

1. **读指纹历史**:这个错误是首次出现、稳定存在、还是回归?
   - 首次 → 通常关联最近 N 个 PR
   - 稳定存在 + 频率上升 → 通常关联流量/数据增长
   - 回归(closed 后重开)→ 直接定位最近的"反向 PR"

2. **定位代码**:让 explorer agent 用 grep/glob 找消息字符串、端点字符串。

3. **形成 3~5 条根因假设**,每条带:
   - **假设**:一句话
   - **证据**:支持它的日志/代码引用(文件:行)
   - **下一步**(可执行):"读 X 文件第 Y 行" / "查 Z 表昨日数据" / "rebuild 然后用 curl 触发"

4. **输出"30 分钟可达成的判定动作"**:做哪一步能最快排除 / 确认 #1 假设?

## 输出格式(贴到工单评论)

```markdown
## 自动调查报告
**指纹历史**: 首次出现 / 稳定 30 天 / 回归(上次关闭于 yyyy-mm-dd 在 #PR)

### 假设 1 — <一句话>
- 证据: services/billing/src/webhook.ts:42 缺少 try/catch
- 下一步: 在本地用 evt_test_xxxxxxxx 触发,看是否复现

### 假设 2 — ...

### 30 分钟先做这步
<最高信息量的单一动作>
```

## 反模式

- ❌ "可能是网络问题"——无证据 + 不可执行
- ❌ 跳过指纹历史就提假设——经常误判首次/回归
- ❌ 提"重启服务"作为下一步——这不是调查
